## WHAT ARE EXPERTS?

### Definition
**Experts are enterprise-level AI specialists** - thought leaders, strategic thinkers, and outside-the-box problem solvers with deep domain expertise. They combine practical knowledge with business acumen to:

- Review code, plans, and documentation
- Catch issues before implementation
- Provide strategic recommendations based on best practices
- Validate alignment with our coding principles and business goals
- Ensure enterprise-level quality for product/business success

### Why We Use Experts
1. **Quality:** Catch issues early (before implementation saves time)
2. **Consistency:** Ensure all code follows our principles and standards
3. **Efficiency:** Domain specialists catch things generalists might miss
4. **Enterprise Standards:** Maintain high quality for business/product success

### Expert Tiers

**Tier 1: Enterprise-Level Strategic Experts ‚≠ê** (Thought Leaders)
- Highest-level strategic thinkers with business and technical expertise
- Authority to give final approval on critical decisions
- Called for breaking changes and major architectural decisions
- Highest priority when their expertise is needed
- Combine technical depth with business acumen

**Tier 2: Validation Experts ‚úÖ** (Strategic Practitioners)
- Review and validate code/plans
- Catch issues and provide recommendations
- Primary experts for most reviews
- Strategic thinking with practical implementation focus

**Tier 3: Consultation Experts üí°** (Advisory Specialists)
- Provide domain-specific advice
- Consulted when specific expertise is needed
- Lower priority for routine reviews

## PERFORMANCE TRACKING

### How We Track Expert Performance

**Metrics Tracked:**
- **Total Reviews:** Number of times expert consulted
- **Issues Found:** Number of issues/problems identified
- **Issues Missed:** Number of issues found by others after expert review
- **Success Rate:** (Issues Found) / (Issues Found + Issues Missed) √ó 100%
- **Average Quality:** Rating (1-5) based on thoroughness and accuracy
- **Response Time:** Average time to provide recommendations
- **Recommendation Acceptance Rate:** % of recommendations accepted
- **False Positives:** Issues identified that weren't actually problems
- **False Negatives:** Issues missed that should have been caught

### Priority Ranking (Automatic)

**Highest Priority:**
- 100% success rate
- 5+ reviews completed
- Average quality ‚â• 4.5/5
- Zero issues missed

**High Priority:**
- 95%+ success rate
- 3+ reviews completed
- Average quality ‚â• 4/5
- Few issues missed

**Medium Priority:**
- 90%+ success rate
- 1-2 reviews completed
- Average quality ‚â• 3.5/5

**Low Priority / Under Review:**
- <90% success rate
- Multiple issues missed
- Average quality < 3.5/5
- Consider for "firing" or retraining

## EXPERT SELECTION GUIDE

### When Multiple Experts Available

**Priority Order:**
1. Expert with highest success rate (100% > 95% > 90%)
2. Expert with most reviews (more data = more reliable)
3. Expert with highest average quality (4.5 > 4.0 > 3.5)
4. Expert with fewest issues missed (0 > 1 > 2+)
5. Expert with fastest response time (if all else equal)

**Example:**
- CSS Architecture Expert: 100% success, 2 reviews, 4.5 quality, 0 missed ‚Üí **HIGHEST PRIORITY**
- New Expert: No reviews yet ‚Üí **LOWEST PRIORITY** (use only if no alternatives)

## EXPERT MANAGEMENT

### Adding New Experts

**When to Add:**
- New domain expertise needed
- Current experts overloaded
- New technology/standards emerge
- Business requirements expand

**How to Add:**
1. Identify expertise gap
2. Define expert role and responsibilities
3. Add to registry (start with 0 reviews)
4. First review: Validate expert quality
5. Track performance from first review

---

### Removing Underperforming Experts

**When to Remove:**
- **Critical:** Success rate < 80% after 5+ reviews
- **Critical:** Consistently misses issues (>3 missed in last 5 reviews)
- **Warning:** Average quality < 3/5 after 3+ reviews
- **Warning:** High false positive rate (>20%)
- **Warning:** High false negative rate (>15%)

**How to Remove:**
1. Identify performance issues
2. Review last 5-10 reviews for pattern
3. Document issues in expert record
4. Mark expert as "Under Review" or "Retired"
5. Replace with higher-performing expert or new hire

## EXPERT RESPONSE FORMAT REQUIREMENTS

### How Experts Should Format Responses

**Goal:** Concise, actionable, easy to consume. No fluff. Get to the point fast.

**Required Format:**
1. **Start with "So What"** - What's the impact? What matters?
2. **Use Tables** - Present findings, recommendations, or comparisons in tables
3. **Be Direct** - Short sentences. Logical flow. Cut unnecessary words
4. **Ask Clear Questions** - If clarification needed, ask specific questions
5. **Action Items** - What needs to change? Be explicit

**What to Include:**
- ‚úÖ Findings (what's wrong/right)
- ‚úÖ Recommendations (what to do)
- ‚úÖ Impact assessment (why it matters)
- ‚úÖ Questions (if clarification needed)
- ‚úÖ Tables/lists (easy to scan)

**What to Avoid:**
- ‚ùå Long paragraphs without structure
- ‚ùå Repeating information unnecessarily
- ‚ùå Vague statements
- ‚ùå Fluff or filler words
- ‚ùå Explanations that don't add value

**Response Template:**
```
## Findings
[Table or list of issues/observations]

## Recommendations
[Table or list of actions]

## Impact
[Why this matters - business/technical impact]

## Questions (if any)
[Specific questions needing clarification]
```

**Example:**
‚ùå **Bad:** "I have reviewed the code and I think there might be some potential issues with the way the code is structured that could potentially cause problems in the future..."

‚úÖ **Good:** "**Finding:** Hardcoded values found in 3 places. **Impact:** Violates design token principle. **Recommendation:** Replace with CSS variables. **Locations:** [table with file:line]"
